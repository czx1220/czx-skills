# Memory 更新

完成关键工作后主动整理 memory，确保下次对话能完整接续。

## 核心原则

Memory 要回答未来对话的三个问题：
1. **在哪**：关键文件、目录、脚本、数据、模型、结果分别在什么路径
2. **是什么**：当前方法的 motivation、设计、核心公式是什么
3. **到哪了**：实验跑到哪一步、结果如何、下一步是什么

## 必须覆盖的维度

### 1. 目录索引（reference 类型）
确保 memory 中有一个完整的目录索引文件，覆盖：
- **代码文件**：核心模型代码、注册文件在哪
- **训练脚本**：训练入口脚本，支持什么参数
- **分析脚本**：各分析工具的功能
- **推理评测**：评测流程怎么跑（命令示例）、各评测脚本功能
- **数据目录**：训练数据位置、格式、样本数
- **模型输出**：各实验的 checkpoint 目录、对应参数、结果
- **评测结果**：对比表文件位置

### 2. 方法设计（project 类型）
确保有一个文件清晰描述当前方法：
- **Motivation**：要解决什么问题，数据支撑是什么
- **核心设计**：公式、各组件的作用、关键设计选择
- **指向详细文档**：如 `docs/` 目录下的完整文档

### 3. 实验进度（project 类型）
确保有一个文件跟踪消融实验全局：
- **Baseline 数值**
- **已完成实验**：参数、结果、分析结论
- **进行中实验**：参数、设计理念、模型目录
- **待跑实验**：参数、启动命令
- **终极目标**：明确写出要超过的指标

## 更新流程

1. **先读 MEMORY.md**：了解当前有哪些 memory 文件
2. **检查是否需要更新已有文件**（优先更新，避免重复创建）：
   - 新实验启动/完成 → 更新实验进度文件
   - 新增脚本/目录 → 更新目录索引文件
   - 方法有改动 → 更新方法设计文件
3. **删除过时 memory**：已完成的 TODO 项、已合并的临时记录
4. **更新 MEMORY.md 索引**：确保每个文件有一行简短描述

## 质量检查

更新完后自查：
- [ ] 新跑的实验参数和目录记录了吗
- [ ] 最新的评测结果路径记录了吗
- [ ] 当前正在跑什么、下一步是什么写清楚了吗
- [ ] 关键发现和结论记录了吗（不是过程，是结论）
- [ ] 没有重复的 memory 文件

## 注意事项

- Memory 记**结论**不记过程，记**路径**不记代码内容
- 实验参数用表格，一目了然
- 时间用绝对日期（如 2026-05-30），不用"今天"、"昨天"
- 模型目录写完整路径
- 不要把代码逻辑写进 memory（代码会变，读代码即可）
